# Signup Data Completeness & Chapter Enforcement
> Date: 2026-03-23
> Status: Approved
> Branch: security-hardening

---

## Problem

Three related gaps in the signup pipeline were identified:

1. **Chapter not enforced** — `profiles.chapter_id` is nullable in the DB and labeled "optional" in the UI, but every member must belong to a chapter for event filtering, officer scope, and QR check-in to work correctly.

2. **Signup data silently discarded** — The `handle_new_user()` trigger (last updated in migration `20260318_rewards_engine.sql`) creates the profile row immediately on `auth.users` INSERT but only writes `id`, `email`, `full_name`, `role`, `spendable_points`, and `lifetime_points`. It does not read `username`, `chapter_id`, or `school_or_company` from `raw_user_meta_data`. When `ensureProfile()` runs after email confirmation, it hits a `23505` unique violation (row already exists via `ON CONFLICT (id) DO NOTHING`), falls back to `fetchProfileById`, and returns the trigger-created row — which has `NULL` for those three fields even though the user provided them at signup.

3. **No avatar on signup** — The `avatars` storage bucket requires an authenticated session (RLS: `auth.uid()`). With email confirmation enabled, there is no session at signup time, so avatar upload is not possible during the form submit step.

---

## Decisions

- **Avatar at signup**: deferred to `OrganizerCodeGate` (the first authenticated screen after email confirmation). Upload is non-blocking and optional.
- **Existing null `chapter_id` rows**: reassigned to Manila (first chapter) in the migration. Safe for dev accounts; no production users exist yet.
- **`ensureProfile` fix strategy**: Approach C — fix the trigger as the primary fix, plus a targeted patch in `ensureProfile` as a safety net for edge cases (e.g. Google OAuth users whose metadata may differ).
- **Points handling**: The `trg_award_signup_bonus` trigger (defined in `20260318_rewards_engine.sql`) handles the 100pt signup bonus AFTER INSERT on profiles. `handle_new_user()` must not award points itself — doing so would cause a double award.
- **Existing dev accounts with `NULL` username**: out of scope for migration. These users will be prompted to set a username the first time they visit ProfileEdit (which already requires username in its Zod schema).

---

## Current Schema State (relevant columns)

```
profiles:
  id, full_name, email, username (nullable),
  school_or_company (nullable), chapter_id (nullable → will be NOT NULL),
  role, avatar_url (nullable), spendable_points, lifetime_points,
  referral_code (auto-generated by trg_profile_referral_code),
  pending_role, pending_chapter_id, created_at
```

`total_points` was renamed to `spendable_points` in `20260318_rewards_engine.sql`. Any trigger referencing `total_points` will fail.

## Design

### 1. Database Migration (`20260323_signup_data_fix.sql`)

Wrap in a transaction to ensure atomicity:

```sql
BEGIN;

-- Step 1: Reassign null chapters to Manila
UPDATE profiles
SET chapter_id = (SELECT id FROM chapters WHERE name = 'Manila' LIMIT 1)
WHERE chapter_id IS NULL;

-- Step 2: Enforce NOT NULL
ALTER TABLE profiles ALTER COLUMN chapter_id SET NOT NULL;

COMMIT;
```

**Note:** The `BEGIN/COMMIT` block above covers only the `UPDATE` and `ALTER TABLE` statements. The `CREATE OR REPLACE FUNCTION` and trigger DDL below runs outside that block as standard PostgreSQL DDL (auto-committed), consistent with the pattern used in `20260318_rewards_engine.sql`.

**Step 3 — Replace `handle_new_user()` trigger:**

Key rules:
- Use `spendable_points` and `lifetime_points` (NOT `total_points` — that column no longer exists)
- Do NOT award points — `trg_award_signup_bonus` handles that AFTER INSERT on profiles
- Fall back to Manila's chapter UUID if `chapter_id` is NULL or unparseable in metadata
- Keep `ON CONFLICT (id) DO NOTHING` as a safety net

```sql
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS trigger
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_chapter_id uuid;
BEGIN
  -- Parse chapter_id from metadata; fall back to Manila if absent or invalid
  BEGIN
    v_chapter_id := (NEW.raw_user_meta_data->>'chapter_id')::uuid;
  EXCEPTION WHEN others THEN
    v_chapter_id := NULL;
  END;

  IF v_chapter_id IS NULL THEN
    SELECT id INTO v_chapter_id FROM chapters WHERE name = 'Manila' LIMIT 1;
  END IF;

  INSERT INTO public.profiles (
    id,
    email,
    full_name,
    username,
    school_or_company,
    chapter_id,
    role,
    spendable_points,
    lifetime_points
  ) VALUES (
    NEW.id,
    NEW.email,
    COALESCE(NEW.raw_user_meta_data->>'full_name', split_part(NEW.email, '@', 1)),
    NEW.raw_user_meta_data->>'username',
    NEW.raw_user_meta_data->>'school_or_company',
    v_chapter_id,
    'member',
    0,
    0
  )
  ON CONFLICT (id) DO NOTHING;

  RETURN NEW;
END;
$$;
```

### 2. `ensureProfile` Safety Net (`useAuthStore.ts`)

**Change `signUp()` function signature**: `chapter_id` becomes a required `string` parameter (not `string | undefined`). This provides compile-time enforcement — TypeScript strict mode will catch any call site that omits it.

**Change `uploadAvatar()` in the store**: Add a file size guard (10 MB limit) to `uploadAvatar()` directly, not just in the calling component. This ensures the limit is enforced consistently from both `OrganizerCodeGate` and `ProfileEdit` (which also calls `uploadAvatar`). Throw a typed error message `'File must be under 10 MB'` so components can display it as-is.

**Change `ensureProfile`**: After a `23505` conflict, instead of returning the trigger-created row as-is, fetch it and patch only fields that are still `NULL`:

```
try INSERT with all signup fields
→ success: return new profile
→ 23505 conflict:
    existing = fetchProfileById(userId)
    patch = {}
    if existing.username is null AND meta.username is set → patch.username = meta.username
    if existing.chapter_id is null AND meta.chapter_id is set → patch.chapter_id = meta.chapter_id
    if existing.school_or_company is null AND meta.school_or_company is set → patch.school_or_company = meta.school_or_company
    if patch is non-empty:
      UPDATE profiles SET patch WHERE id = userId
      return { ...existing, ...patch }
    else:
      return existing
```

On every subsequent login the existing row will already be fully populated, so the patch will be empty and the UPDATE is skipped.

**Note on `spendable_points` / `lifetime_points` in the INSERT path**: `ensureProfile` currently inserts these as `0` — this is correct, since `trg_award_signup_bonus` fires AFTER INSERT and sets them to 100.

### 4. Avatar Picker on OrganizerCodeGate (`OrganizerCodeGate.tsx`)

**Placement**: top of the card, above the existing heading. **Shown only in the normal state (code input form), not in the `submitted` state** (success card with checkmark + "Continue as Member" button). Once the user reaches the submitted state, the avatar section is hidden.

**Navigation safety**: if an upload is in progress when the user taps "Request Officer Access" or "Continue as Member", navigation proceeds immediately — the upload is fire-and-forget. Avatar URL will be set asynchronously; a missing avatar is never a blocker.

**UI:**
- Circular avatar (`w-16 h-16`) using initials placeholder (same style as the rest of the app)
- Camera icon (`<Camera />`) overlay at bottom-right of the circle
- Tapping opens a hidden `<input type="file" accept="image/*">`
- **File size limit: 10 MB** — if the selected file exceeds this, show inline error "Image must be under 10 MB"
- On valid file select → show preview immediately (local object URL via `URL.createObjectURL`)
- Upload via `useAuthStore.uploadAvatar()` → on success call `updateProfile({ avatar_url })` to persist
- During upload: spinner replaces the camera icon on the circle; preview image remains visible
- On upload failure: show inline error below the circle ("Upload failed — tap to retry"); camera icon returns

**Error states** (all three are surfaced by the store's `uploadAvatar` throwing; the component catches and displays):
1. File type not allowed → "Only image files are allowed"
2. File too large (> 10 MB) → "Image must be under 10 MB" *(enforced in `uploadAvatar()` store method)*
3. Upload failure → "Upload failed — tap to retry"

**Avatar is entirely optional** — no validation, no required state, no blocking of the next step.

## Implementation Order

1. Apply and verify the DB migration (`20260323_signup_data_fix.sql`) first — trigger fix + NOT NULL constraint
2. Implement `SignUp.tsx` Zod change + label change
3. Update `useAuthStore.ts` — `signUp()` signature, `ensureProfile` patch, `uploadAvatar` size guard
4. Implement avatar picker in `OrganizerCodeGate.tsx`

The `ensureProfile` patch (step 3) is a safety net for edge cases. Apply and verify the trigger fix (step 1) independently before relying on the patch to catch failures.

## Out of Scope

- Client-side image compression / resizing (Supabase Storage handles large files within the 10 MB limit)
- Avatar picker for returning users who already have `avatar_url` set (they use ProfileEdit)
- Backfilling `username` for existing dev accounts with `NULL` username — these users will be prompted on first visit to ProfileEdit (which requires username in its Zod schema — this is intentional)
- Google OAuth signup (currently behind a `<ComingSoonModal />`) — the Manila fallback in the trigger future-proofs this path
